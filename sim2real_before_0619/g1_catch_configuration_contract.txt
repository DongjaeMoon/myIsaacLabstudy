# G1 Humanoid Catching Project — Real/MuJoCo/IsaacLab Configuration Contract
# Last updated: 2026-05-19
# Purpose:
#   This file freezes the system configuration and I/O contract for the Unitree G1
#   AprilTag box catching project.
#
#   It is intended to be given to Codex/AI before writing IsaacLab training code.
#   The training environment MUST match this contract so that the exported policy.pt
#   can be inserted into sim2real/configs/g1_catch_real.yaml and run immediately
#   through sim2real/g1_catch_real.py.

================================================================================
0. PROJECT GOAL
================================================================================

Final real-world experiment:

    A real Unitree G1 humanoid stands in a catch-ready posture.
    A human lightly tosses an AprilTag-attached delivery box toward the robot.
    The robot detects the box using camera/AprilTag tracking and catches/holds it
    using a learned whole-body policy.

Important design philosophy:

    We design the real/MuJoCo deploy system FIRST.
    Then IsaacLab training must be designed backward from that deploy contract.

Execution chain:

    IsaacLab training
        -> export TorchScript policy.pt
        -> insert policy path into sim2real/configs/g1_catch_real.yaml
        -> run the SAME sim2real/g1_catch_real.py in MuJoCo via DDS loopback
        -> run the SAME sim2real/g1_catch_real.py on the real G1

There should be no separate MuJoCo-only controller.

1. TOP-LEVEL SYSTEM ARCHITECTURE
================================================================================

Main deploy entrypoint:

    sim2real/g1_catch_real.py

Main configuration file:

    sim2real/configs/g1_catch_real.yaml

Important modular files:

    sim2real/config.py
        Compatibility wrapper that re-exports config loader/schema.

    sim2real/catch_real/controller.py
        High-level controller orchestration:
        - mode switching
        - scripted pose transitions
        - autonomous policy running
        - status logging
        - safe shutdown

    sim2real/catch_real/policy_runner.py
        Loads TorchScript policy and converts policy action to target q:
        - clipped_action
        - q_target_raw = q_ref + action_scale * clipped_action
        - optional policy-step target delta clamp
        - LPF:
              q_target_filtered =
                  (1 - alpha) * previous_policy_target
                  + alpha * q_target_raw

    sim2real/catch_real/obs_builder.py
        Builds the deploy observation vector.

    sim2real/catch_real/object_observation.py
        Current object observation provider/stub:
        - zeros by default
        - optional fake object debug toggle
        - later replaced/extended by AprilTag estimator

    sim2real/catch_real/motor_command.py
        Writes LowCmd q/dq/kp/kd/tau/crc.

    sim2real/catch_real/state_reader.py
        Reads motor q/dq and IMU state from LowState.

    sim2real/catch_real/dds_io.py
        DDS publisher/subscriber initialization and LowState callback.

    sim2real/catch_real/modes.py
        Mode enum.

    sim2real/catch_real/keyboard.py
        Terminal keyboard input.

    sim2real/catch_real/safety.py
        Target clamping, stale state checks, damping behavior, etc.

Debug/demo entrypoint:

    sim2real/g1_catch_config_demo.py

    This is only for simple pose/mode verification using the same config/controller
    structure. The real deploy file is still sim2real/g1_catch_real.py.

2. DDS / LOW-LEVEL COMMUNICATION
================================================================================

Communication backend:

    Unitree DDS

MuJoCo:

    Use the SAME deploy code with DDS loopback:

        --net-iface lo

Real G1:

    Use the SAME deploy code with the real network interface:

        --net-iface <REAL_INTERFACE_NAME>

DDS topics:

    LowState input topic:
        rt/lowstate

    LowCmd output topic:
        rt/lowcmd

Message type:

    hg

Low-level control frequency:

    control_dt = 0.002 s
    low-level command loop = 500 Hz

Policy frequency:

    policy_dt = 0.020 s
    policy inference loop = 50 Hz

Motion command type:

    Joint position PD command through LowCmd.

For each controlled motor:

    q      = q_des
    dq     = dq_des, usually 0
    kp     = YAML-defined per-joint kp
    kd     = YAML-defined per-joint kd
    tau    = 0
    mode   = 1

The policy NEVER outputs torque directly.

================================================================================
4. POLICY INPUT / OUTPUT CONTRACT
================================================================================

Policy file format:

    TorchScript policy.pt
    Loaded with torch.jit.load(...)

Required policy input/output:

    input:
        obs shape = [batch, 104]

    output:
        action shape = [batch, 29]

Policy action meaning:

    action is residual joint position target, not torque.

Control law:

    clipped_action = clip(action, -action_clip, +action_clip)

    q_target_raw = q_ref + action_scale * clipped_action

    q_target_filtered =
        (1 - target_lowpass_alpha) * previous_policy_target
        + target_lowpass_alpha * q_target_raw

    q_des = q_target_filtered

    dq_des = 0
    tau_ff = 0

Current LPF setting:

    runtime.target_lowpass_alpha = 0.2

Interpretation:

    alpha = 1.0  -> no low-pass filtering
    alpha = 0.2  -> moderate smoothing
    alpha = 0.1  -> stronger smoothing, slower reaction

Important:

    LPF is applied only to policy-generated targets.
    Scripted pose transitions are not filtered by policy LPF.
    previous_policy_target is reset when entering autonomous policy mode.

================================================================================
5. ACTOR OBSERVATION CONTRACT
================================================================================

Total actor observation dimension:

    obs_dim = 104

Observation order MUST be exactly:

    1. projected_gravity
        dim = 3
        source = IMU orientation
        frame = policy body frame

    2. base_ang_vel
        dim = 3
        source = IMU gyro
        frame = policy body frame

    3. joint_pos_rel
        dim = 29
        value = q - q_ref
        joint order = controlled joint order in Section 3

    4. joint_vel
        dim = 29
        value = dq
        joint order = controlled joint order in Section 3

    5. previous_action
        dim = 29
        value = previous policy action
        order = action order in Section 3

    6. object_rel_pos
        dim = 3
        value = box position relative to robot body/base/policy frame
        source = AprilTag estimator in real system
        source = simulator-derived but AprilTag-like signal in IsaacLab

    7. object_rel_lin_vel
        dim = 3
        value = box linear velocity relative to robot body/base/policy frame
        source = AprilTag estimator in real system
        source = simulator-derived but AprilTag-like signal in IsaacLab

    8. tag_visible
        dim = 1
        value = 0 or 1

    9. mode_one_hot
        dim = 4
        mode names = safe_stand, catch_ready, catch, hold
        total mode_one_hot dim MUST remain 4

Total:

    3 + 3 + 29 + 29 + 29 + 3 + 3 + 1 + 4 = 104

Do NOT include in actor obs:

    - true world-frame object state
    - privileged object mass/friction/restitution
    - contact forces
    - simulator-only contact history
    - exact base linear velocity unless later proven real-estimable
    - torque obs, unless the deploy shell is also changed

Current deploy convention when no object is visible:

    object_rel_pos     = [0, 0, 0]
    object_rel_lin_vel = [0, 0, 0]
    tag_visible        = 0

When fake object debug is toggled:

    tag_visible        = 1
    object_rel_pos     = [0.8, 0.0, 0.4]
    object_rel_lin_vel = implementation-defined fake incoming velocity
                          if configured in object_observation.py

================================================================================
6. MODES / POLICY RUNTIME
================================================================================

Runtime modes:

    DAMPING
        Sends damping commands.

    SAFE_STAND
        Scripted safe standing pose.

    CATCH_READY
        Scripted catch-ready pose.
        This is the posture from which autonomous policy operation begins.

    CATCH
        Reinterpreted as AUTONOMOUS POLICY RUNNING mode.
        This is NOT a one-time manual catch trigger anymore.
        The policy runs continuously at 50 Hz in this mode.

    HOLD
        Scripted hold pose.

Important final experiment behavior:

    The real experiment should NOT depend on pressing K at the exact toss time.

Correct scenario:

    1. Robot moves to catch_ready.
    2. Robot enters autonomous policy running mode.
    3. Policy runs continuously.
    4. If no box is visible:
           tag_visible = 0
           object_rel_pos = 0
           object_rel_lin_vel = 0
           policy should learn to maintain ready/balance.
    5. When AprilTag box appears and approaches:
           tag_visible = 1
           object_rel_pos/object_rel_lin_vel become nonzero
           same running policy reacts and catches.
    6. After catch, policy or scripted logic should result in stable holding.

Keyboard controls:

    S:
        safe_stand scripted pose

    P or C:
        catch_ready scripted pose

    A:
        autonomous policy running mode
        internally uses catch mode

    K:
        manual/debug policy running mode
        same internal behavior as A, but used only for debugging

    O:
        toggle fake object observation for debugging

    H:
        hold scripted pose

    D:
        damping

    Q or ESC:
        safe shutdown / damping exit

Potential future setting:

    policy_runtime.auto_start_after_ready = false currently.
    Later it may be set true so that the robot automatically enters autonomous
    policy mode after reaching catch_ready.

8. ISAACLAB TRAINING REQUIREMENTS
================================================================================

Training must match deploy.

Training policy type:

    Continuous autonomous waiting-and-catching policy.

Not event-triggered:

    Do NOT train a policy that only runs after an external "catch" button.
    The policy must run continuously after ready.

Episode structure:

    1. Reset robot near catch_ready pose.
    2. During waiting phase:
           no box visible or tag_visible = 0
           object_rel_pos = 0
           object_rel_lin_vel = 0
           policy must keep stable ready posture.
    3. At random toss time:
           box appears / is tossed toward robot
           tag_visible becomes 1
           object_rel_pos/object_rel_lin_vel are provided.
    4. Policy reacts to catch.
    5. Reward holding/stability after impact.

Randomization required:

    - random waiting time before toss
    - random toss position
    - random toss velocity
    - random box mass/size/friction/restitution if desired
    - AprilTag-like visibility dropout/noise
    - object relative position/velocity noise
    - mild robot state noise/domain randomization

Actor observation in IsaacLab:

    Must be exactly Section 5.
    Must be 104 dim.
    Must be ordered exactly as deploy.

Critic observation:

    Can include privileged simulator information.
    Actor must not.

Action in IsaacLab:

    Must be exactly 29 dim.
    Must be residual joint position action.
    Must use same joint order as Section 3.
    Must use same action_scale as YAML.

Control in IsaacLab:

    The action-to-target conversion used during training must match deploy:

        q_target = q_ref + action_scale * action

    q_ref should be catch_ready reference pose during autonomous policy mode.

Policy frequency:

    50 Hz equivalent.
    IsaacLab env dt and decimation must produce policy_dt = 0.020 s.

Recommended reward components:

    Waiting/stability:
        - maintain upright posture
        - maintain catch_ready reference
        - small joint velocity
        - small action magnitude
        - small action rate
        - foot stability / no foot slip if free-standing

    Catching:
        - object close to torso/arms catch zone
        - reduce object relative velocity after contact
        - keep object from falling
        - successful hold duration

    Safety:
        - penalize large lower-body actions
        - penalize excessive waist/ankle movement
        - penalize falling
        - penalize unstable base orientation

Important real-world simplification:

    The first successful real experiment should be:
        ready posture -> light toss -> catch -> hold for a few seconds

    Walking/carrying is later.
    Locomotion is currently out of scope for this deploy shell.

9. POLICY EXPORT / INJECTION PROCEDURE
================================================================================

After IsaacLab training:

    Export TorchScript policy.pt with:

        input  = [1, 104]
        output = [1, 29]

Before putting it into YAML:

    Test shape locally:

        python - <<'PY'
        import torch
        path = "/absolute/path/to/policy.pt"
        policy = torch.jit.load(path, map_location="cpu")
        policy.eval()
        x = torch.zeros(1, 104)
        with torch.no_grad():
            y = policy(x)
        print(y.shape)
        assert tuple(y.shape) == (1, 29)
        PY

Put policy path into YAML:

    policy:
        path: /absolute/path/to/policy.pt

or run with CLI override:

    python sim2real/g1_catch_real.py \
        --config sim2real/configs/g1_catch_real.yaml \
        --net-iface lo \
        --use-policy \
        --policy /absolute/path/to/policy.pt \
        --start-pose catch_ready

    python sim2real/g1_catch_real.py \
        --config sim2real/configs/g1_catch_real.yaml \
        --net-iface lo \
        --use-policy \
        --start-pose catch_ready

Then:

    A -> autonomous policy running
    O -> fake object observation toggle for debug

11. COMMON FAILURE MODES
================================================================================

If policy.pt fails with matrix shape error:

    The policy was probably trained with wrong obs_dim.
    Old policies may be obs_dim=141.
    Current deploy contract requires obs_dim=104.

If robot moves wrong joints:

    motor_indices/action_order/joint_order mismatch.
    Check Section 3 and YAML action_order.

If policy works in IsaacLab but not in MuJoCo/real:

    Usually one of:
        - obs order mismatch
        - joint order mismatch
        - action_scale mismatch
        - q_ref mismatch
        - policy_dt mismatch
        - body frame mismatch for gravity/object rel state
        - old privileged observation leaked into actor

If robot jitters:

    Check:
        - zero policy jitter
        - kp/kd too high
        - target_lowpass_alpha too high
        - action_scale too large
        - action rate too high
        - policy output saturation

Current small jitter with zero/dummy policy is considered acceptable for now,
but learned policy must still be tested conservatively.
